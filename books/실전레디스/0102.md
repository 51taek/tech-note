# 0102: 레디스의 시작 & 자료형과 기능

### 레디스를 왜 쓰는가? (레디스의 장점)

- 레디스는 인메모리 데이터베이스이기 때문에 처리속도가 아주 빠르다. → 성능의 이유로 레디스를 쓴다.
- 운영체제에서 외부 라이브러리를 따로 설치하지 않아도 되도록 구성함.
- 다양한 자료형
- 데이터 영속성
- 레플리카, 클러스터링을 통한 고가용성
- 요청-응답 통신
- 루아 스크립트를 이용한 처리
    - 한번에 여러개의 레디스 명령을 사용할 수 있음 (원자성 보장)
- 싱글 스레드
    - 레디스 6.0 이후로 데이터 접근은 싱글 스레드지만 I/O는 멀티스레드 활성화 가능

### 레디스의 주요 자료형

- String
    - key-value 형태
    - 값에 숫자를 넣고 숫자를 증가시키거나 감소시키는 걸 할 수 있음. (자세한건 찾아봐야함)
    - 한번에 여러 키 값을 읽어오거나 저장할 수 있음
    - 512MB 제한이 있는데 설정으로 더 늘릴 수 있다.
- List
    - key : ["user3", "user2" ] 형태
    - 일반적인 List에서 가능한 명령어는 대부분 지원하는듯하다.
    - 조회, 삽입 모두 O(N)
- Hash
    - key : {"name": "원택", "age": "24", "job": "backend"} 형태
    - hash내 각 필드에는 TTL 설정이 불가능
    - 값이 숫자인 경우에 사용할 수 있는 명령어가 있음
    - 모든 값 반환 O(N), 하나만 반환 O(1)
    - 하나의 해시에 들어가는 요소가 많으면 메모리 압축을 제대로 활용하지 못함 → 차라리 String형 쓰는게 나음
- Set
    - 리스트와 비슷하지만 중복이 없음
- Sorted Set
    - Set인데 순서가 있음
    - 탐색, 삽입이 O(logN),

### 레디스의 보조 자료형/기능

- 비트맵
    - 실제로 String 형으로 정의되어 있음
    - 메모리 공간을 효율적으로 다룰 수 있음. But, 처리하는 숫자가 적다면 Set이 더 효율적
- 지리 공간
    - 주소나 경도, 위도 나타냄
    - 내부적으로 Sorted Set으로 저장되어 있음
    - 위도/경도를 합쳐서 GeoHash 라는 명령어 제공
- Pub/Sub
    - 발행자가 수신자에게 메시지를 정보를 저장하지 않고 메시지를 보냄
    - 새로 채널을 구독할 때 이전에 발행된 메시지는 볼 수 없음
    - 네트워크 장애로 인해서 채널 접속이 중단되어 재시도 접속해도 중간에 유실된건 날라감
    → List형에도 데이터를 유지할 수 있도록 해야함
        - 레디스 5.0 이후라면 레디스 스트림을 사용할 수 있음
    - 구독자가 메시지를 받아서 처리를 못하는 경우 레디스 측의 부하가 커질 수 있음
        - Pub/Sub용 클라이언트 출력버퍼로 해당 클라이언트에 제한을 설정해 대상 클라이언트의 연결 해제 가능
    - 레플리카를 사용하는 경우 마스터에서 발행된 메시지는 레플리카에도 복제됨
        - 레플리카에 발행된 메시지는 다른 노드에 영향을 주지 않고 그 노드를 구독하고 있는 구독자에게 메시지를 전달
        - 메시지를 마스터에 명령어로 전달한 경우 반환값은 해당 노드를 구독하고 있는 구독자 수가 된다
    - 레디스 7.0 미만에서 레디스 클러스터를 사용하는 경우, 노드에 발행된 메시지는 다른 모든 노드에 전달되므로 모든 노드에서 구독됨
        - 높은 처리량이 필요한 Pub/Sub 워크로드를 처리할 때 클러스터 버스의 버퍼 관리로 인해 메인 스레드 CPU 부하가 높아질 수 있음
    - → Redis Sharded Pub/SUb 이라는 게 나옴. 메시지 반영이 샤드 내로 제한됨. 키가 특정 슬롯에 할당되는 것처럼 샤드 채널도 슬롯에 해당됨
- HyperLogLog
    - 방문자 수 처럼 대략적인 숫자만 알아도 될 때 사용
    - 정확한 수치는 틀릴 수 있음. But, 메모리 공간을 효율적으로 활용 가능
    - 
- 레디스 스트림
    - 카프카와 비슷한 기능이지만 카프카는 메시지를 읽는 소스에서 파티션으로 분할하는 반면, 레디스 스트림은 미리 준비된 Consumer의 상태를 기반으로 논리적인 파티션을 구현한다. 카프카와 비슷하게 하기 위해서는 key를 여러개 두어서 유사한 파티셔닝을 할 수 있다
    - Redis Pub/Sub은 네트워크가 중단됐을 때 중간에 유실된 메시지를 받지 못하지만 레디스 스트림에서는 이를 놓치지 않을 수 있음
    - 레디스 5.0 이후에 추가
    - Consumer Group 쿼리 모드에서는 하나 이상의 Consumer에서 스트림 내 엔트리를 분산시키면서 처리할 수 있다 (카프카의 Shared Group과 유사한 것 같음)

내부 인코딩

- String 형
    - int: 64비트 부호 정수
    - embstr: 44바이트 이하 문자열
    - raw: 44바이트 보다 큰 문자열
- List 형
    - 양 끝의 매개변수에 따라 특정 조건에서는 LZF로 압축하지 않은 상태. 이외에는 LZF로 압축한 상태
- Hash 형
    - listpack: 특정 조건을 만족할 때만 인코딩
    - hashtable: 위에꺼 아닐때
- Set 형
    - intset: 모두 정수일때
    - hashtable: 위에꺼 아닐때
- Sorted Set 형
    - listpack: 특정 조건
    - skiplist: 위에꺼 아닐 때
- HyperLogLog 형
    - Sprase: 특정 조건에 렌 렝스 부호화를 이용해 압축
    - Dense: 위에꺼 아닐 떄
- 지리적 공간 인덱스
    - GeoHash를 이용해 52비트

레디스의 데이터 삭제

레디스에는 키를 삭제하는 명령어로 DEL과 UNLINK 두 종류가 있다

수백만개의 요소를 포함한 키라면 삭제하는데 수초가 걸릴 수 있음. 이 시간동안 DEL로 동기 처리하면 블록됨

UNLINK에는 Lazy Freeing이라는 기능이 있음

키가 먼저 백그라운드 큐에 넣어지고 다른 스레드에 의해 비동기 및 순차적으로 삭제됨. 

→ 비동기적으로 삭제되기 때문에 정합성 이슈에 대해서 생각해봐야 함.
