레디스 서버는 메모리를 다음과 같은 형태로 사용

- 데이터 및 데이터 구조 오버헤드
- 클라이언트 쿼리 버퍼
- 클라이언트 출력 버퍼
- AOF 버퍼
- AOF 재작성 버퍼
- 레플리케이션 백로그
- 레디스 함수
- 이페머럴 스크립트
- 메모리 단편화

INFO MEMORY 명령어를 통해서 레디스 서버의 메모리 내역을 자세하게 확인할 수 있다

레디스는 malloc 함수로 메모리를 할당하기 때문에 운영체제의 메모리 페이지 할당 맾이을 제어할 수 없도록 구현되어 있다

따라서 레디스에서 데이터 삭제 명령어를 실행해도 확보된 메모리가 운영체제로 반환된다는 보장이 없으며 실제 메모리 사용량인 RSS도 변하지 않는다

레디스 클라이언트에서 보낸 명령어는 레디스 서버의 클라이언트 쿠리 버퍼 라는 공간에 일단 저장한 후 소켓을 통해 데이터를 점진적으로 읽는다

읽어오는 데이터가 크거나 복잡한 명령어인 경우 버퍼에서 모든 처리 내용을 모아서 버퍼링한다

버퍼의 크기는 예전에는 1.0으로 고정되어 있었지면 4.0.7 이후부터는 client-query-buffer-limit 지시자를 통해서 변경할 수 있게되었다

명령어의 실행 결과는 클라이언트 출력 버퍼라는 클라이언트별 버퍼에 저장된다

레디스는 점진적으로 소켓에 데이터를 저장하는 방식을 통해서 데이터를 원자적으로 처리한다

클라이언트 출력 버퍼 크기를 초과하여 메모리를 사용하면 끊어진다

하드리미트와 소프트 리미트가 있다

### INFO Memory 출력 결과 해석

user_memory는 레드스가 메모리 할당자를 사용하여 할당한 메모리의 크기를 의미하고 데이터 이외에 클라이언트 출력 버퍼 등도 포함된 크기이다.

레플리카 출력 버퍼와 AOF 버퍼 크기는 used_memory에 추가되지만 maxmemory를 계산할 때는 포함되지 않는다

레디스 7.0 이후부터는 mamemory-clients 지시자를 통해 모든 클라이언트가 사용하는 메모리 소비량을 제한할 수 있고 메모리 소비량이 제한값을 초과하면 소비량이 가장 많은 클라이언트의 연결이 해제된다

기본적으로 클러스터 링크의 전송용 버퍼의 크기는 제한이 없다

Pub/Sub 기능으로 사용하는 메시지로 인해 메모리 사용량이 큰 경우, cluster-link-sendbuf-limit 값을 client-query-buffer-limit 값보다 크게 설정하여 대처할 수 있다

### 메모리 단편화 문제 대처 방법

단편화율이 1.0이거나 약간 더 높으면 양호. 1.5 이상이면 단편화 심각. 1.0 이하면 스왑 발생

1. 레디스 재시작
2. 동적 단편화
3. 메모리 스왑 제한
4. 메모리 할당자 변경
5. MEMORY PURGE 명령어 실행
    1. 메모리 사용량이 높은 최대 사용량 시간에도 jemalloc이 메모리를 즉시 해제하지 않는 상황에서 메모리를 재사용하고자할 때 유용

### 클라이언트 출력 버퍼

레디스 서버는 클라이언트마다 COB(Client Output Buffer)를 갖추고 서버에서 처리한 결과를 직접 클라이언트에게 보내지 않고 일단 COB에 저장. 그 후 COB에 저장된 정보가 클라이언트에 한번에 전송

Pub/Sub 기능을 사용할 때 COB에 제한을 두어 메시지를 소비하지 못하는 클라이언트 연결을 끊을 수 있다

#### replica 유형의 COB

마스터에서 버퍼링하는 영역으로, 레플리케이션 과정 중에 처리된 마스터의 쓰기 작업을 레플리카에도 반영하기 위해서 사용됨

스냅숏 생성 시 이 과정에서 캐시 노드 내의 데이터를 직렬화하여 데스크에 저장

이 기간에는 마스터 변경 로그를 레플리카용 COB에 저장한 후 스냅숏 생성이 완료되면 변경 로그를 레플리카로 전송

스냅숏 생성 작업은 레플리카에서 수행하는 것이 가장 좋음. 하지만 이 과정에서 레플리카의 COB에 쓰기 요청이 쌓이므로 주의가 필요함

#### normal 유형

일반적인 클라이언트 연결에 사용되는 버퍼 영역

많은 클라이언트에서 하나의 명령을 보낸 후 다음 명령을 보내기 전에 응답이 올 때 까지 기다리는 블로킹 방식을 사용

→ 일반 클라이언트의 연결을 끊는 것으 바람직하지 않은 경우가 많기 때문에 normal 유형의 COB에 대한 제한값은 보통 0임

### 키 만료

#### 만료 방법

크게 수동적 방법과 능동적 방법이 있고 기본적으로 둘을 조합해서 사용

- 수동적 방법은 클라이언트가 키에 접근할 때 해당 키의 TTL이 만료되었는지 확인하고 만료되었다면 해당 키를 메모리에서 삭제
- 능동적 방법은 레디스가 매 초마다 10번, 임의로 20개의 키를 샘플링하여 만료된 키를 확인하고 삭제
    - 샘플링된 키 중 25% 이상이 만료된 경우 다시 샘플링과 삭제 과정을 반복

#### 삭제 정책

메모리 사용량이 maxmemory 지시자로 설정된 값에 도달했을 때 메모리 확보를 위해 키를 삭제하는 과정을 함

옵션을 통해서 어떤 키들을 삭제하거나 삭제하지 않을지에 대한 정책 설정을 할 수 있음(LFU or LRU)

### 동적 리해싱

레디스는 하나의 큰 해시 테이블을 사용하여 데이터를 관리함

동적 리해싱 방식은 이 해시 테이블의 크기를 사용하려는 상황에 맞추어 자동으로 조정하는 기능

리해싱 과정에서는 크기가 조정된 새로운 테이블을 추가로 준비하여 새로운 아이템이 이 테이블에 저장되도록 함

각 작업을 수행될 때마다 오래된 테이블에서 새 테이블로 데이터가 옮겨지고 데이터가 모두 이동하면 이전 테이블은 삭제됨

### 동적 단편화 제거

jemalloc이 그래도 단편화가 덜 발생하도록 설계되어있어서 이걸 사용하는게 좋음

동적 단편화 제거 방법을 사용하면 재시작 없이도 단편화 문제 해결 가능

특정 임계값을 초과했을 때 jemalloc의 기능을 사용하여 연속된 메모리 영역에 새롱누 데이터 레플리카를 생성한 후 오래된 데이터를 해제하는 방식으로 진행

레디스 4.0에서 추가되었고 5.0에서 버전 2로 업데이트
