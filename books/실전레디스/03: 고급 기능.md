

## 파이프라인

여러 명령어를 동신에 송신하여 RTT(왕복시간)를 줄일 수 있음

원자적 처리를 보증할 수 없기 때문에 트랜잭션이나 루아스크립트를 통해서 따로 보장해야함

파이프라인 내에서는 순서가 보장되기 때문에 여러 명령을 한번에 하고 응답을 나중에 한꺼번에 하는게 효율적임

## 루아

레디스의 내장 스크립트 언어

조건 분기와 같은 복잡한 로직 구현 가능

쓰기와 읽기 시간을 최소화 하도록 설계

레디스 7.0 이후로 레디스 함수로 루아 스크립트 실행 가능. 이전에는 이페머럴 스크립트로 실행.

### 이페머럴 스크립트

여러 요소를 동시에 처리할 수 있음

루아 디버거가 있음

레디스 클러스터는 인수로 지정할 키 값을 기준으로 라우팅하기 때문에 스크립트 내에 사용할 키를 인수로 지정하는 것이 좋다. (Redis Cluster에서 Lua Script를 실행할 때, 스크립트 안에서 사용할 key를 Redis가 미리 알아야 한다)

EVAL 명령어는 오버헤드가 커서 보통 EVALSHA를 많이 사용한다

💡이퍼메럴 스크립트에는 롤백이 구현되어 있지 않기 때문에 완벽한 원자성을 보장하지는 않는다

EVAL이나 EVALSHA로 수행한 스크립트는 삭제 명령을 실행하지 않는 한 캐시로 남아있음 ← 스크립트 실행 속도를 높이기 위해서

남아있는 스크립트가 메모리를 잡아먹는다는 사소한 문제(그렇게 용량이 크지 않음)가 있는데 레디스 함수는 이런 문제를 개선했음

레디스 7.0 이후부터 이페머럴 스크립트에 플래그 기능을 사용할 수 있음

루아 디버깅에 —ldb와 -ldb-sync-mode가 있음

—ldb 옵션을 활성화 한 상태에서 변경한 작업은 롤백됨. —ldb-sync-mode를 활성화 한 상태에서 변경한 작업은 롤백 되지 않음

#### 단점

- 디버깅이 어려움
- 스크립트를 그대로 렌더링 하는 안티패턴을 사용할 확률이 높음
- 스크립트 간 코드 공유와 재사용이 어려움
- 루아 5.1만 사용 가능

### 레디스 함수

레디스 7.0의 대표적인 기능으로 이페머럴 스크립트의 단점을 개선하기 위해서 나옴 → 쓸 수 있다면 이페머럴 대신 레디스 함수를 쓰는 것이 좋음

원자적 처리가 보장됨

실행중에는 다른 처리를 블록하기 때문에 장시간 실행하면 안됨

함수가 마스터에서 레플리카로 복제됨

스냅샷이나 AOF로 영속화 되어 데이터와 동등한 내구성을 가짐

라이브러리 라는 개념을 도입하여 코드 공유 간소화

다양한 언어 엔진을 사용할 수 있으며 추후에 자바스크립트와도 호환이 예정되어 있음. 현재는 루아 5.1만

#### 루아에 대해서

레디스 서버가 동작하는 호스트에 접근 할 수 없도록 샌드박스 환경에서 실행됨

레디스 서버 내에 동작하는 스크립트에서 레디스 명령어를 호출하기 위해서 레디스 API를 활용함

레디스 루아 API 레퍼런스 참고하면 됨

레디스 명령어를 사용할 때는 redis.call()과 redis.pcall()을 사용할 수 있다

redis.call()은 에러가 발생하면 스크립트가 중단되고 레디스 서버가 예외를 반환한다. 하지만 redis.pcall()은 스크립트를 바로 중단시키지 않고 예외를 반환하도록 한다. (ex try-catch)

## 트랜잭션

레디스 트랜잭션은 롤백 기능을 제공하지 않고 명령어 실패 시 그대로 남은 작업을 수행한다

MULTI 명령어로 시작해서 큐에 쌓았다가 EXEC 명령어가 실행되면 한번에 순서대로 실행함

WATCH 명령어를 통해서 낙관적 락을 구현할 수 있음

## 모듈

레디스 소스코드를 수정하지 않고도 C로 구현하여 레디스 서버에 독립적으로 추가할 수 있다

모듈을 통해 새로운 자료형 추가 가능

레디스 모듈은 레디스 서버 데이터에 접근하기 위해 고수준 API와 저수준 API를 제공한다

저수준 API는 레디스 네이티브 명령만큼 빠르다

## 클라이언트 캐싱

클라이언트가 로컬에 캐시를 저장하면 다른 클라이언트가 캐싱된 키를 업데이트 할 때 데이터 정합성이 깨진다.

이를 위해서 레디스는 ‘무효화 테이블’ 이라는 하나의 전역 테이블에 캐싱된 키를 갖는 클라이언트의 목록을 기록해둔다

만약 해당 키가 업데이트된 경우 저장된 클라이언트에게 무효화 메시지를 전송하고, 메시지를 받은 클라이언트는 해당 키의 캐시를 무효화한다

브로드캐스트 방식을 쓰면 특정 키에 대한 접근을 기록하지 않는다
