### 비동기 처리를 통한 구현

레디스의 레플리케이션은 이벤트트 루프의 비동기 논블로킹 방식으로 구현됨

비동기 동작으로 인해 지연이 발생하는 경우, 마스터와 레플리카의 데이터가 동일하다가 보장할 수 없음

### 레플리케이션을 사용할 때 레디스를 연결하는 방법

각 캐시 노드의 IP 주소나 엔드포인트를 기록해두고 레디스 클라이언트에서 직접 접근하거나 twemproxy 같은 프록시를 활용해 여러 레플리카의 요청 라우팅 과정을 자동화

### 기본적으로 읽기 전용인 레플리카

레플리카에 직접 쓰기 작업을 수행할 수도 있지만, 문제 발생 위험을 고려해야함

기본적으로는 레플리카에 쓰기작업 불가능하도록 설정되어있음

### 마이그레이션으로 활용

새로운 레디스 서버로 마이그레이션할 때 레플리케이션 활용 가능

### 레플리케이션 주의사항

레플리케이션 작업을 실행할 때 마스터에 영속성이 설정되어 있지 않은 경우 엔진을 재실행하거나 종료하면 데이터 세트가 초기화됨. 레플리카도 초기화됨

### 레플리케이션을 시작할 때 메커니즘

1. 레플리카는 PSYNC 명령어로 마스터에 연결을 요청하며, 해당 시점까지 처리한 레플리케이션 ID와 오프셋을 전송
2. 마스터는 요청받은 마스터의 레플리케이션 ID와 자신의 레플리케이션 ID가 일치하는지 확인하고, 오프셋 레플리케이션 백로그의 버퍼에 있는지 확인
3. 요청받은 오프셋이 레플리케이션 백로그에 있는지에 따라 처리가 달라짐
    1. 레플리케이션 백로그에서 동기화가 가능한 경우 요청된 오프셋이 레플리케이션 백로그에 있고, 레플리케이션 백로그에서 동기화가 가능한 경우에는 부분 동기화를 실행
    2. 레플리케이션 백로그에서 동기화가 불가능한 경우 레플리케이션 중단 중에 마스터가 받은 쓰기 작업 요청의 크기가 버퍼 크기를 초과하여 부분 동기화가 불가능한 경우에는 전체 동기화 실행

전체 동기화가 요청 처리에 미치는 영향이 아주 크기 때문에 조건을 만족하면 부분 동기화를 할 수 있도록함

전체 동기화는 프로세스가 포크처리되어 마스터의 메모리 데이터를 덤프하고 디스크에 RDB파일을 저장

마스터에서 덤프한 모든 데이터를 전송하여 네트워크 대역폭을 압박할 수도 있음

### 전체 동기화

1. 레플리카가 마스터에 레플리케이션 시작을 요청
2. 마스터는 BGSAVE 명령어를 실행하여 프로세스를 포크 처리하고 포크된 프로세스에서 메모리 스냅숏을 진행
3. BGSAVE 처리가 완료된 후, RDB 파일을 레플리카로 전송. 그동안 마스터의 쓰기작업은 레플리카의 클라이언트 출력 버퍼에 기록됨. 레플리카는 전송된 RDB 파일을 메모리로 불러옴
4. RDB 파일 전송이 완료된 후 레플리카의 클라이언트 출력 버퍼에 기록된 데이터를 레플리카로 전송
5. 레플리카의 클라이언트 출력 버퍼에 쓰기 작업이 완료되면, 마스터의 쓰기 작업은 실시간으로 레플리카로 계속해서 전송

#### 세부사항

마스터가 RDB 파일을 전송할 떄 이미 레플리카에 RDB 파일이 있으면 기존꺼는 삭제함

전체 동기화를 할 때 중간에 클라이언트가 레플리카에 요청을 하면 레플리카는 기본적으로 응답을 보냄. 이러면 동기화 이전의 데이터가 반환될 수 있음.

replica-serve-stale-data를 No로 변경하면 오류 메시지를 응답하게 할 수 있음

#### 레플리카가 여러개 있을 때

BGSAVE 처리가 완료되기 전에 두번째 레플리카 이후 동기화를 요청할 경우 해당 레플리카는 첫번째 레플리카와 동일한 내용으로 복제

BGSAVE가 완료된 이후 요청하는 경우에는 전체 동기화 과정이 처음부터 시작

#### 레플리케이션 연결이 끊길 때

레플리카의 클라이언트 출력 버퍼가 초과되면 연결이 끊기고 재동기화를 시작한다

부분동기화를 할 수도 있고 전체 동기화를 할 수도 있다.

레플리카 클라이언트의 출력 버퍼는 client-output-buffer-limit으로 설정. replica 클래스를 지정하여 값을 설정. 기본으로는 256mb이며 하드리밋이라 초과되면 바로 즉시 연결종료하고 64mb가 넘어가면 소프트리밋으로 60초 후 연결 종료한다

#### TTL이 있는 키의 레플리케이션

레플리카는 TTL이 있는 키를 만료시키지 않고 마스터에서 만료시킨 후에 마스터에서 레플리카로 DEL 명령을 전달함

RDB 파일을 덤프할 때 만료된 키는 복제하지 않으므로 RDB 파일을 읽을 때 마스터에 존재했던 만료된 키는 복원되지 않음

마스터에서는 만료된 키가 회수될 때까지 INFO나 DBSIZE 명령어를 통해 키를 인식하지만 레플리카에는 키가 적재된지 않음

#### 디스크 없는 레플리케이션

원래 디스크 백업 레플리케이션은 전체 동기화 과정에서 프로세스가 포크되어 마스터의 메모리 데이터를 RDB로 덤프하여 디스크에 쓰는 방식

디스크 없는 레플리케이션은 프로세스가 포크된 후 RDB 파일을 직접 소켓으로 전송

이후 추가된 레플리케이션은 진행중인 작업 기다려야함

큰 데이터 세트를 처리하는 경우나 네트워크 대역폭이 크고 디스크 속도가 느린 경우에 유용함

현업에서도 많이 사용

#### 그 외 최적화

레플리카 소켓의 TCP_NODELAY 활성화: 지연시간은 감소하지만 대역폭 증가

### 부분 동기화

레플리케이션 연결이 끊어진 동안의 모든 쓰기 작업이 레플리케이션 백로그에서 레플리카로 전송

백로그의 크기는 repl-backlog-size로 설정할 수 있으며 기본은 1MB

레플리케이션의 연결이 끊어진 후 일정 시간이 지나면 백로그가 해제되도록 동작(repl-backlog-ttl)

### 동기식 레플리케이션

WAIT 명령어를 통해서 할 수 있지만 좋은 방식은 아닌듯함.

비동기 레플리케이션의 특성 상 레플리카에 데이터가 제대로 저장되지 않을 수 있음

### 마스터와 레플리카의 연결 상태 모니터링

마스터가 10초 간격으로 핑을 보냄

레플리카도 마스터로 핑을 보냄

### 페일오버

레디스는 기본적으로 페일오버 기능이 없음

레디스 센티넬이나 레디스 클러스터 기능을 사용해 고가용성 보장
