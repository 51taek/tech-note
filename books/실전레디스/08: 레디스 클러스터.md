### 레디스 클러스터의 장점

- 샤딩을 제공
- 레디스 클러스터에 읽기 확장성을 추가하고 싶다면 READONLY 명령어로 레플리카에서 읽기 쿼리를 수행하도록 강제할 수 있다
- 클라이언트의 요청에 대해 해당 노드로 분산시키는데 프록시를 사용하지 않기 때문에 프록시로 인한 오버헤드가 없음
- 데이터를 가지고 있지 않은 노드에 요청이 들어올 경우 클라이언트에게 데이터를 가지고 있느 마스터의 위치를 알려주고 해당 노드로 요청을 리다이렉트
    - 이 과정에서 얻은 정보를 클라이언트가 저장하면 리다이렉트에 의한 오버헤드도 줄일 수 있음
- 최종적 일관성을 채택

### 레디스 클러스터가 사용하는 2개의 TCP 포트

1. 클라이언트로부터 연결을 받는 포트 (6379)
2. 클러스터 내부 통신을 위한 포트 (기본값에 10,000을 더한 값) ← 클러스터 버스 포트

NAT나 포트 포워딩을 사용하는 도커와 같은 컨테이너 환경에서는 마스터에서 보이는 레플리카의 IP주소나 포트 번호가 다름

→ 클러스터 내 다른 노드에게 자신의 캐시 노드 IP 주소와 클러스터 버스 포트 번호를 임의의 값으로 설정할 수 있도록 되어있음

클러스터 버스 내 통신에는 이진 프로토콜이 사용됨

노드는 완전 메시 구조로 구성되며 노드 간에는 구성 정보, 상태와 같은 정보 교환

페일 오버 검증, 설정 업데이트 등에 사용되는 하트비트 패킷 교환도 이뤄짐. 가십 프로토콜을 통해서.

Raft 라는 분산 합의 알고리즘 기반

### 동작 매커니즘

1. 클라이언트가 클러스터를 구성하는 캐시 노드 중 하나의 IP 주소에 접속
2. 접근 노드의 레플리카 마스터인 경우 접근 분기
    1. 접근 노드가 레플리카 인 경우
        1. 읽기 쿼리 : 
            1. 클라이언트 설정에 따라 READONLY 명령어 실행 여부에 따라 다름
                1. 요청 된 캐시 노드에서 요청 처리
            2. 그 외
                1. 레디스 서버가 MOVED 리다이렉트를 클라이언트에 응답
                2. 클라이언트가 MOVED 리다이렉트를 받고 로컬 슬롯 매핑을 업데이트
                3. 클라이언트가 해당 키의 슬롯을 가진 샤드에 접근
        2. 쓰기 쿼리
            1. 레디스 서버가 MOVED 리다이렉트를 클라이언트에 응답
            2. 클라이언트가 MOVED 리다이렉트를 받고 로컬 슬롯 매핑을 업데이트
            3. 클라이언트가 해당 키의 슬롯을 가진 샤드에 접근
    2. 접근 노드가 마스터
        1. 키가 해당 캐시 노드의 슬롯 범위 내
            1. 요청된 마스터에서 처리
        2. 키가 해당 캐시 노드의 슬롯 범위 밖
            1. 레디스 서버가 MOVED 리다이렉트를 클라이언트에 응답
            2. 클라이언트가 MOVED 리다이렉트를 받고 로컬 슬롯 매핑을 업데이트
            3. 클라이언트가 해당 키의 슬롯을 가진 샤드에 접근

### 장애 탐지

마스터에 문제가 발생하면 마스터의 레플리카 중 하나를 승격시킴

#### 캐시 노드의 상태(PFAIL,FAIL)

PFAIL

- Possible Failure
- 특정 노드가 다른 노드에 핑을 보내고 타임아웃 이내에 핑이 오지 않을 때 로컬에서 해당 노드를 PFAIL로 표시
- 타임아웃 / 2 시간만큼 응답이 오지 않으면 연결 재시도

FAIL

- FAIL 상태가 되어야 페일오버 실행 가능
- 과반수의 마스터 노드가 대상 노드를 PFAIL 혹은 FAIL 상태로 판단하는 경우 해당 노드는 FAIL 상태가됨
- 판단하는데 cluster-node-timeout * cluster-replica-validity-factor 만큼 시간 소요

레디스 클러스터의 페일오버는 노드의 과반수가 정상동작하고 있는 것을 전제로 하기 때문에 대부분의 노드가 장애시 동작하지 않음

오픈 소스 버전의 클러스터는 소규모의 부분적인 장애 복구가 목표이므로 대규모의 장애 복구에는 적절하지 않음

이러한 동작 방식 때문에 네트워크 장애 시 데이터가 손실 됨 위험이 있음

#### 레플리카 선출

다음 조건을 만족해야함

1. 레플리카 마스터가 FAIL
2. 마스터가 하나 이상의 슬롯 관리
3. 레플리카가 일정 시간 이상 마스터와 연결 끊김

마스터 선출 과정

1. 마스터의 FAIL을 감지한 레플리카가 일정 시간동안 대기 후 클러스터 내의 각 마스터에게 FAILOVER_AUTH_REQUEST 요청을 브로드캐스트함
2. 각 마스터는 패킷을 받으면 투표로 응답함
3. 레플리카는 currentEpoch 이하의 응답을 무시하고 나머지는 반영
4. 새로운 마스터가 된 노드는 다른 마스터보다 더 크게 configEpoch를 증가시킴

### 해시태그

어떤 키에 값을 저장할 때는 해당 키 이름을 해싱하여 어느 슬롯에 데이터를 저장할지 결정한 후 해당 슬롯을 가진 샤드에 데이터가 저장됨

레디스 클러스터는 데이터의 일관성과 높은 성능을 제공하기 위해 MSET과 같이 여러 키를 동시에 조작하는 명령이나 이페머럴 스크립트를 실행할 때 모든 키가 동일한 슬롯에 있어야 함

데이터를 저장할 때는 해시태그 기능을 사용하며, 키가 달라도 같은 슬롯에 접근할 수 있게 된다

해시태그를 남용하면 슬롯 간 요청에서 편향이 발생할 수 있다

해시태그의 조건

- 키에 ‘{’ 를 포함해야 한다
- ‘{’의 오른쪽에 ‘}’를 포함해야 한다
- {} 사이에 하나 이상의 문자가 포함되어야 한다

### 클러스터 버스

클러스터 내의 각 노드는 클러스터 버스라고 하는 TCP 버스를 통해 이진 프로토콜과 완전 메시 구조로 서로 연결되어 있음

레디스 7.0부터는 클러스터 통신을 위해 대기하는 포트 번호를 cluster-port 지시자로 설정할 수 있음

가십 프로토콜로 노도의 상태를 파악하고 Raft라는 분산 합의 알고리즘을 기반으로 각 노드의 설정 정보를 공유

클러스터 내의 각 노드가 서로 연결되어 있어도 모든 노드 쌍에 핑을 주고 받는 완전 메시 방식을 사용하지는 않음

CLUSTER MEET <ip> <port> 명령을 보내면 해당 명령을 ㅂ다은 노드는 대상 노드를 클러스터의 일부로 인식

노드 A가 다른 노드 B에 대해 노드 B가 알지 못하는 클러스터 내 다른 노드 정보를 가지고 있을 때 노드 A에서 노드 B로 보내는 가십 메시지를 통해 노드 B도 해당 노드를 인식하게 함

하트비트 메시지는 핑퐁을 보낼 때 항상 슬롯 구성의 정보를 포함하여 전달함

수신자가 발신자의 노드 정보가 오래되었다고 판단하면 UPDATE 메시지로 업데이트를 요청

### 파티셔닝

파티셔닝의 3가지 방법

- 클라이언트 측 파티셔닝
    - 클라이언트에서 어느 노드로 요청을 보낼 지 결정
- 프록시 기반 파티셔닝
    - 프록시가 클라이언트의 요청을 받아 대상 노드로 라우팅
- 쿼리 라우팅
    - 클라이언트의 요청을 받았을 때 처음에는 무작위 노드로 보내고 그 노드가 요청을 처리하기에 적합하지 않으면 대상 노드로 요청을 리다이렉트

레디스는 클라이언트와 쿼리 라우팅을 결합한 형태

슬롯 매핑 정보는 클라이언트가 시작할 때나 MOVED 리다이렉트가 발생했을 때 로컬 슬롯 맵이 업데이트 되지만 ASK 리다이렉트의 경우 로컬 슬롯 맵은 업데이트 되지 않음
