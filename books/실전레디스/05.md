## 캐시 서버로서 레디스 아키텍처

### 읽기 관점 아키텍처

#### 지연 로딩 패턴(Cash Aside 패턴)

어플리케이션이 레디스에 정보를 요청하고 캐시 미스나면 데이터베이스에 물어보는 방식.

사용하지 않는 데이터를 캐시로 저장하는 공간이 작음. 메모리 관점에서 효율적.

#### Read-Through 패턴

어플리케이션이 레디스에 정보를 요청하고 캐시 미스 시에 레디스가 데이터베이스에게 최신 캐시를 받아오는 방식.

### 쓰기 관점 아키텍처

#### Wirte-Trough 패턴

쓰기 작업때 시간이 조금 증가해도 데이터베이스에 업데이트할 때 캐시 서버에도 같이 업데이트 하는 방식.

데이터 전체가 최신 데이터를 반영하고 있을 수 있음

#### Write-Back 패턴

쓰기 작업때 캐시에 먼저 저장 후 스케줄링으로 데이터베이스에 값을 업데이트 하는 방식.

쓰기 오버헤드를 줄일 수 있음.

#### Write-Around 패턴

쓰기 작업 때 데이터베이스에 먼저 반영하고 그걸 캐시에 반영하는 방식.

데이터 손실의 위험이 낮음.

### 멀티스레드

레디스는 기본적으로 싱글스레드로 처리하지만 부분적으로 멀티스레드를 활용할 수 있다.

레디스 6.0 이상에서 옵션으로 I/O 작업을 멀티스레드로 처리할 수 있다.

1. 먼저 메인 스레드로 연결 요청을 받음
2. 소켓을 생성하고 읽기 작업을 대기하는 큐에 정보를 넣음
3. 메인스레드에서 읽기 이벤츠 처리를 마친 후 I/O 스레드에 순서대로 작업을 할당
4. 메인스레드는 I/O
