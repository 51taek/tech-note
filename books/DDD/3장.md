## 🧭 도메인 복잡성 관리

같은 단어를 두 가지 분야에서 서로 다른 의미로 사용할 때 비즈니스 도메인 모델에 문제가 생긴다.

---

### 🧱 바운디드 컨텍스트 (Bounded Context)

유비쿼터스 언어를 여러 개의 작은 언어로 나눈 다음,  
각 언어를 적용할 수 있는 명시적인 바운디드 컨텍스트에 할당하면 된다.

같은 단어여도 다른 바운디드 컨텍스트에 있다면 의미가 완전히 달라진다.  
바운디드 컨텍스트는 서로 영향을 주지 않는다.

![image.png](attachment:b06a61f7-3fa0-4b43-876f-cd4a7c46d414:image.png)

유비쿼터스 언어는 바운디드 컨텍스트 경계 안에서만 보편적으로 적용된다.

바운디드 컨텍스트는 하나의 모델이다.  
모델은 어떤 목적을 가지고 그 정보를 나타내는 데 의의가 있다. (ex. 지도)

바운디드 컨텍스트의 크기는 정답이 없다.  
다만 모델 자체로 유용해야 한다.

큰 바운디드 컨텍스트에서 세분화된 바운디드 컨텍스트를 추출하는 이유에는  
새로운 소프트웨어 엔지니어링 팀을 구성하거나 시스템의 일부 비기능 요구사항을 해결하는 것이 포함된다.

같은 데이터에서 작동하는 응집된 유스케이스의 집합을 식별하되,  
그것을 여러 개의 바운디드 컨텍스트로 분해하지 마라.

---

### ⚖️ 바운디드 컨텍스트 vs 하위 도메인

하위 도메인은 **발견하는 것**이고,  
바운디드 컨텍스트는 **설계하는 것**이다.

바운디드 컨텍스트와 하위 도메인이 **일대일 관계를 맺는 것은 항상 정답이 아니다.**

---

### 🚧 경계 (Boundary)

바운디드 컨텍스트는 모델 경계뿐만 아니라  
이를 구현하는 **시스템의 물리적 경계 역할**도 한다.

각 바운디드 컨텍스트는 **개별 서비스나 프로젝트로 구현**되어야 한다.

바운디드 컨텍스트는 여러 하위 도메인을 포함할 수 있다.  
이러한 경우 바운디드 컨텍스트는 **물리적 경계**,  
하위 도메인은 **논리적 경계**다.  
(논리적 경계는 네임스페이스, 모듈, 패키지 등을 뜻한다.)

---

### 💡 시맨틱 도메인 (Semantic Domain)

**어떤 특정 분야의 지식, 개념, 관계를 정의하는 시맨틱(의미론적) 기술이 적용된 영역**을 의미한다.

‘모니터’, ‘포트’, ‘프로세서’라는 단어는  
소프트웨어와 하드웨어 엔지니어링 시맨틱 도메인에서 서로 다른 의미를 갖는다.
